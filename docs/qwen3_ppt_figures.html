Qwen3.5 算子拆分与替代尝试

patch_qwen35_linear_attn_mul.py · ONNX 图级改写 · 最终仍失败

triangular_update 核心死锁

来源：chunk_gated_delta_rule 内部
for index in range(1, attn.shape[-1]):
    row = attn[..., index, :index].clone()
    sub = attn[..., :index, :index].clone()
    attn[..., index, :index] = row + \
        (row.unsqueeze(-1) * sub).sum(-2)
NPU synchronize() 60s 死锁 — 动态循环边界无法静态化

5D Mul TBE 不支持

来源：linear_attn 子图 · /model/layers.0/linear_attn/Mul
ATC ERROR:
Operator [Mul] in node
  [/model/layers.0/linear_attn/Mul]
  is not supported
尝试 Reshape(5D→4D) + Mul + Reshape — 单算子可绕过，整体仍失败

INT64 Mul TBE 不支持

来源：形状计算子图
INT64 × INT64 → TBE 编译失败
替代：Cast(→INT32) + Mul + Cast(→INT64)
单独可绕过，但无法解决核心 triangular_update 问题

Scalar Mul / Add 广播 TBE 不支持

来源：linear_attn 缩放与偏置
Scalar × Tensor → Expand + Mul
Add(...,1) broadcast → Tile + Add
绕过隐式广播限制，但底层算子仍阻塞
算子替代流程图
5D Mul → Reshape(5D→4D) + Mul(4D) + Reshape TBE 不支持 5D 浮点乘法
INT64 Mul → Cast(→INT32) + Mul + Cast(→INT64) TBE 不支持 INT64 乘法
Scalar × Tensor → Expand(scalar→shape) + Mul TBE 不支持标量隐式广播
Add (...,1) 广播 → Tile(dim→match) + Add TBE 不支持尾维广播
Identity Expand → 直接删除节点，重连输入输出 输入输出 shape 相同时为空操作
最终结论：即使在 ONNX 图层面逐一替代了不兼容算子，chunk_gated_delta_rule 展开后的 triangular_update 动态循环仍无法被 ATC 静态图编译器处理。Qwen3.5 hybrid Mamba 架构在 310B1 上不可部署。

4 种实验模式结果
模式操作发现
hooks Layer0 各子模块注册 forward pre-hook 打印到 linear_attn 入口后 → 无输出
manual 单独执行 embed_tokens + layer0.forward layer0.forward 挂起
delta_steps 逐步跟踪 chunk_gated_delta_rule 内部 卡在 triangular_update 阶段

诊断辅助证据：NPU npu-smi 显示挂起期间 AICore=0%，表明 Da Vinci 核未执行任何 kernel；推断 hang 发生在 AICPU（Host 侧 CPU 算子）或 NPU 调度层。日志中出现 "The oprator of add is executed, Currently High Accuracy but Low Performance OP with 64-bit"，说明部分算子已 fallback 到 64-bit Host 侧执行路径。

动态图 vs 静态图

为什么 SSM 算子无法在 Ascend 310B1 NPU 上运行

⚡ 动态图 (Eager Mode)

  • 执行方式：逐行解释 Python 代码，每条算子即刻发射到硬件
  • 控制流：支持任意 Python if / for / while
  • 循环边界：可依赖运行时张量值
  • 优化程度：无全局优化，逐算子执行
  • 代表框架：PyTorch (默认), TensorFlow Eager
灵活 无法整体优化
VS

⚙ 静态图 (Graph Mode)

  • 执行方式：编译器一次性编译整个计算图为固定执行计划
  • 控制流：所有分支和循环必须编译期确定
  • 张量形状：必须编译期已知或声明为有限动态维度
  • 优化程度：算子融合、内存复用、流水调度
  • 代表框架：ATC/TBE, TensorRT, XLA
高效推理 不支持动态控制流
冲突核心：数据依赖的动态控制流
# Qwen3.5 的 triangular_update — 循环边界依赖张量维度 for index in range(1, attn.shape[-1]): # ← 动态边界 row = attn[..., index, :index].clone() # ← 每次切片范围不同 sub = attn[..., :index, :index].clone() attn[..., index, :index] = row + (row.unsqueeze(-1) * sub).sum(-2)
ATC 路径失败：
TBE 编译器要求所有 shape 编译期已知。遇到 linear_attn 的 5D Mul 和动态切片 → Operator not supported
torch_npu 路径失败：
虽名义上支持 eager，但底层仍将算子分发到 NPU 图引擎。动态循环产生的大量小 kernel 在异步队列中形成数据依赖死锁
为什么 Qwen3-0.6B 能成功？
标准 GQA Transformer：所有算子（MatMul、Softmax、RMSNorm、SiLU 等）shape 固定、无动态循环，完美适配静态图编译。28 层结构规整，ATC 一次性编译为 .om 离线模型，NPU Da Vinci 核高效流水执行。

结论

Ascend 310B1 的静态图架构是其高效 AI 推理的基础，但也意味着含 SSM/Mamba 动态循环的混合架构当前无法直接部署。团队经过系统化排查验证了这一兼容性边界，最终选择标准 Transformer 架构（Qwen3-0.6B）绕过限制，成功在边缘 NPU 上运行 LLM。

这一底层探索虽然以失败告终，但精确定位了不兼容算子（triangular_update、chunk_gated_delta_rule），为后续 CANN 版本升级或模型适配提供了明确的方向。

项目价值

成功验证：Qwen3-0.6B 通过 PyTorch → ONNX → ATC → .om → ACL 全链路部署于 Ascend 310B1 NPU，PPL=22.44，65.3 tok/s，证明了边缘 NPU 运行 LLM 的可行性。

底层探索：经系统化调研证明 Qwen3.5 混合 Mamba 架构因 SSM 算子（triangular_update 动态循环、chunk_gated_delta_rule）与 310B1 静态图不兼容。团队尝试了 5 种算子级 ONNX 图改写均无法绕过核心限制。

技术决策：基于以上发现，团队选择标准 GQA Transformer 绕过限制，这一决策有充分的工程验证支撑。
